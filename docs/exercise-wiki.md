# AWS Blocks — "Ask my AWS account" exercise: Q&A / Wiki notes

Reference notes captured while building a read-only "ask my AWS account" chat agent
on **AWS Blocks** (Public Preview, 0.1.x), with tool reach provided by the
**AWS API MCP server**. Written as a learning log — each entry is a question that
came up while building, with the answer distilled for future reference.

> Stack: AWS Blocks (`@aws-blocks/*`) · Strands Agents SDK · AWS API MCP server
> (`awslabs.aws-api-mcp-server`) · React + Vite · target region ap-northeast-1 (Tokyo).

**Companion pages:** [Glossary — acronyms & technical terms](./glossary.md) ·

## 1. AWS Blocks fundamentals

### What is `bb-agent`?

`bb` = **Building Block**. Every AWS Blocks package is named `bb-<thing>`
(`bb-agent`, `bb-kv-store`, `bb-realtime`, `bb-auth-basic`, …). So `bb-agent` is
"the Agent Building Block."

A **Building Block** is a managed-service-style abstraction *defined in your
application code* (CDK underneath). One constructor call expands into a pile of
provisioned AWS resources **plus** the runtime code that drives them — and it runs
locally with mocks, then deploys to the real services unchanged.

`new Agent(scope, 'x', {...})` alone provisions:

| Resource | Service | Purpose |
|---|---|---|
| Conversations + messages tables | DynamoDB ×2 | Conversation history (persistence) |
| Session snapshots | S3 | Strands agent state between turns |
| Streaming channel | API Gateway WebSocket | Push tokens to the browser live |
| Async execution | SQS + Lambda | Run the agent without API Gateway timeouts |

The agent loop itself (deciding when to call a tool, threading history) is the
**Strands Agents SDK**, which `bb-agent` wraps and wires to the infra.

### `aws-blocks/` vs `src/` — which is backend, which is frontend?

A common mix-up. They are two halves of one repo:

| Directory | What it is | What happens to it |
|---|---|---|
| **`aws-blocks/`** | **Backend** — agent, auth, API, tool handlers, the CDK stack (`index.cdk.ts`) and Lambda handler | **Provisioned to AWS** — becomes DynamoDB, S3, API Gateway, SQS, Lambda |
| **`src/`** | **Frontend** — the React UI (`App.tsx`, `main.tsx`) | **Built** by Vite into static files, then *hosted* (CloudFront/S3) and downloaded by the browser |

Mnemonic: **`aws-blocks/` deploys *as* cloud services; `src/` ships *to* the
browser.** The name "aws-blocks" is a trap — it's the backend+infra folder, not the
UI. They talk over HTTP (locally `:3000` UI → `:3001` backend; in the cloud
CloudFront → API Gateway → Lambda).

### How does it serve a website on localhost — Docker, or plain HTML?

**No Docker, no containers.** `npm run dev` starts two plain **Node.js** processes:

| Process | Port | Role |
|---|---|---|
| Vite dev server | `:3000` | Bundles `src/` and serves the React app to the browser |
| Blocks dev server (`tsx watch`) | `:3001` | Runs the `aws-blocks/` backend with AWS services **mocked in-process** |

The "🔌 Attaching dev server" log lines (for S3 / WebSocket) are *more in-process
Node servers* standing in for cloud services — not Docker images. This is a
deliberate design choice: many local AWS dev stacks (e.g. LocalStack) use
containers; Blocks uses lightweight in-process mocks instead, which is why it boots
in ~1.5s.

## 2. The toolchain

### Vite vs React — what's the difference?

They are not alternatives; they are different layers.

- **React** = the **UI library you write** (components, state, `App.tsx`). Describes
  *what the interface is*.
- **Vite** = the **build tool + dev server** that compiles and serves that React
  code (bundles `.tsx` → browser JS, runs the hot-reloading `localhost` dev server).

Analogy: **React is your application source; Vite is the build pipeline + local web
server that ships it.** (Vite is the faster modern successor to webpack in this role.)

### What is `node_modules/`?

The install target for **every dependency** in `package.json` *plus* all of their
transitive dependencies.

- JS equivalent of a Python venv's `site-packages` / vendored libraries.
- Generated by `npm install` from `package.json` + `package-lock.json` (exact pinned
  versions). Never hand-edited.
- **Disposable and git-ignored** — delete it, run `npm install`, and the lockfile
  rebuilds it identically.
- It's why `import { Agent } from '@aws-blocks/blocks'` resolves: the package code
  physically lives in `node_modules/@aws-blocks/blocks/` (including that package's
  own docs under `node_modules/@aws-blocks/blocks/docs/*.md`).

### What is `uvx` (and `uv`)?

**`uvx` is to Python what `npx` is to Node.** It's part of **`uv`**, a fast
(Rust-built) Python package manager that replaces `pip` + `venv` + `pipx`.

`uvx <pkg>` downloads a Python CLI tool into a throwaway isolated environment and
runs it — no global install, no manual virtualenv, no polluting system Python.

In this exercise the Blocks tool handler launches the AWS API MCP server with:

```bash
uvx awslabs.aws-api-mcp-server@latest
```

It mattered because system Python here was 3.9.6 but that server needs ≥3.10 — `uv`
provisioned a compatible Python itself instead of us juggling versions by hand.

### npm vs CDK — which am I actually running on deploy?

Running `npm run sandbox` (or `deploy`) uses **both**, in layers:

```
npm run sandbox            # npm: runs the "sandbox" script from package.json
  -> tsx sandbox.ts        # a TS script Blocks generated
    -> CDK                 # synthesizes CloudFormation from index.cdk.ts
      -> CloudFormation    # AWS actually creates DynamoDB, S3, Lambda, ...
```

- **npm** = Node's package manager + **script runner**. Its only job here is to run
  the command behind the `sandbox` alias. It knows nothing about AWS.
- **CDK** (AWS Cloud Development Kit) = **infrastructure-as-code**. It's the engine
  that turns your TypeScript infra (`index.cdk.ts` / `BlocksStack`) into
  CloudFormation and deploys it.

You never type `cdk` directly because Blocks wraps it — its `sandbox.ts`/`deploy.ts`
scripts call CDK programmatically. **Tell them apart in the logs:** npm/tsx output at
the start, then CloudFormation stack/resource progress (`CREATE_IN_PROGRESS`) once CDK
takes over — that's the real AWS work.

## 3. Running a real model

### What is Ollama? Who made it? Is it safe?

- **What:** a tool for running LLMs **locally** — download a model once, then it
  serves an API on `localhost` that apps call. Like a tiny local Bedrock; no cloud,
  no per-token cost.
- **Who:** **Ollama Inc.** (founded ~2023 by Jeffrey Morgan and Michael Chiang).
  **Open source, MIT-licensed**, widely adopted.
- **Safe?** Yes, with normal caveats: install from the official source
  (`brew install ollama` / ollama.com). **Privacy upside:** inference runs locally,
  so prompts and AWS data never leave the machine. The real caveat is *quality* —
  small local models (e.g. 8B) are weaker and can make clumsy tool calls; that's a
  correctness issue, not a security one.

In Blocks, local dev uses Ollama if it's running, and **falls back to the canned
mock automatically** if it isn't (the `[canned]` replies seen before installing
Ollama).

### War story: picking a local model size (and getting tool-calling to work)

Running a real local model surfaced two very practical lessons:

1. **Model size vs. hardware — `llama3.1:8b` crashed the machine.** On an Apple M2
   with ~5.3 GiB of GPU memory, loading the 8B model (~4.9 GB weights + runtime
   overhead) plus the OS/IDE/browser caused a memory-pressure crash. **Fix:** drop to
   `llama3.2:3b` (~2 GB), which fits with headroom. Rule of thumb: the model weights
   must leave real headroom under your available (V)RAM, not just barely fit.

2. **Small models are weak at tool-calling.** With 3B, the first attempt at
   "list my S3 buckets" produced a *description of what the API does* — it never
   called the `runAwsRead` tool. The tool was available; the model just didn't choose
   to use it (a classic small-model failure mode: it pattern-matches "explain" instead
   of "act"). **Fix = blunt, directive prompting.** Adding an explicit instruction to
   the system prompt —

   > IMPORTANT — ALWAYS ACT, NEVER DESCRIBE: To answer ANY question about the
   > account, you MUST call the runAwsRead tool and base your answer ONLY on its
   > actual output … For "list my S3 buckets", call runAwsRead with command
   > "s3api list-buckets".

   — plus starting a **fresh conversation** (so the model doesn't copy its own prior
   non-tool answer) got it to actually call the tool and return real bucket names.

3. **Residual limitation.** Even after calling the tool correctly, the 3B narrated
   clumsily (hedged with "this appears to be…", hallucinated a fake account ID). The
   data was right; the prose wasn't. **Takeaway:** small/local/free models are great
   for proving wiring and for privacy, but reliable *and* polished agentic behavior
   needs a more capable model (e.g. Bedrock's Claude at deploy). The
   small-vs-large / local-vs-cloud trade-off is a core applied-AI judgment call.

**Diagnosing tool-calls:** the signal isn't the chat bubble — it's the `[dev:server]`
log line `Attempting to execute AWS CLI command: aws s3 list-buckets …`. If that line
is absent, the model answered without using the tool.

## 4. Safety & architecture

### "Are you creating actual AWS resources?"

During local development — **no.** `npm run dev` runs entirely on in-process mocks.
The only commands that create real AWS resources are `npm run sandbox` and
`npm run deploy` (which run CDK to provision DynamoDB, S3, API Gateway, SQS, Lambda).

The only real AWS calls made during the exercise were **read-only** (e.g.
`sts get-caller-identity`, `s3api list-buckets`). An attempted `s3api create-bucket`
was **refused** — see guardrails below.

### Why "read-only" is enforced in 3 layers (defense in depth)

The system prompt saying "you are read-only" is the **weakest** layer — a courtesy,
not a control. Real enforcement is below it:

1. **`READ_OPERATIONS_ONLY=true`** on the AWS API MCP server — refuses any mutating
   CLI verb at the server. (Proven: `create-bucket` → *"Execution of this operation
   is denied by security policy."*)
2. **Read-only IAM** on the runtime principal (enforced at deploy) — the hard cap.
3. **System prompt** — UX/courtesy only; do not rely on it.

Key lesson: if the model were jailbroken into calling the tool with a mutating
command, layers 1 and 2 still stop it. Prompt instructions are not a security
boundary.

### How does streaming (tokens appearing live) work?

Two resources do two different jobs:

- **SQS + Lambda** = *where the agent runs* — a background Lambda pulled off a queue,
  so a slow multi-turn LLM call doesn't sit inside an HTTP request and time out.
- **API Gateway WebSocket** (the `Realtime` / `bb-realtime` block) = *how results
  get to the browser live* — a persistent two-way connection. The browser subscribes
  once; the background Lambda **publishes** token chunks to the channel; the browser
  receives them as they arrive.

Plain HTTP is one-shot (ask once, answer once); a WebSocket is what lets the server
keep pushing. That's why the client hook subscribes *before* sending, and why
`streamingMode: 'token'` makes tokens appear one at a time.

### Deploy gotcha: the `aws` / `AWS` name prefix is reserved

First `npm run sandbox` failed with:

> `AWS::ResourceGroups::Group ... "Group name must not start with 'AWS'"`

Cause: Blocks creates AWS **ResourceGroups** whose names derive from the **stack
name**, and AWS **reserves the `AWS`/`aws` prefix (case-insensitive)** for
ResourceGroup names. Our stack was `aws-ask-agent-stack-…`, so it was rejected.

- **Fix:** rename so the stack name doesn't start with `aws` — we used
  `ask-aws-agent-stack-…` (and renamed the `Scope` `aws-ask` → `ask-aws` for
  consistency). The **stack name** is the one that mattered (it feeds the
  ResourceGroup name), not the `Scope`.
- **Failure is non-destructive:** CloudFormation rolled the failed stack back to
  `ROLLBACK_COMPLETE` (no live resources), but that empty stack shell must be
  **deleted** before redeploying the same name.

### Tagging every resource in the stack

To tag all taggable resources, add this in `index.cdk.ts` right after `new cdk.App()`:

```ts
cdk.Tags.of(app).add('iac-stack', 'CDK');
cdk.Tags.of(app).add('project', 'aws-block+aws-mcp');
```

`Tags.of(app)` propagates to every taggable construct (DynamoDB, S3, Lambda, SQS…).
Note ResourceGroups themselves aren't tagged, and tag values may use
`+ - _ . : / = @` and spaces.

### Going public safely: an invite-code signup gate

`AuthBasic` has **open self-signup** and no native way to restrict it (only
`passwordPolicy`, `codeDelivery`, `crossDomain`). A public deploy with open signup =
anyone can create an account and run up your Bedrock bill. To gate it:

1. **Don't expose the raw auth API.** Instead of `export const authApi = auth.createApi()`
   (which publishes the public `signUp` endpoint), keep `auth` **server-side only**.
2. **Route auth through your own `ApiNamespace`** methods: `me()`, `signIn()`,
   `signOut()`, and `signUp(inviteCode, …)` which rejects unless the code matches.
3. **Store the code in a secret `AppSetting`** (`secret: true` → encrypted SSM), set
   out-of-band — `aws ssm put-parameter … --type SecureString` in prod, and
   `.bb-data/settings.json` (keyed by the setting's `fullId`) locally. Never in git.
4. **Frontend:** replace the built-in `AccountMenuBar` with a small custom form that
   collects the invite code on signup.

Note: the AWS-account-reading tool stays **stubbed** in the cloud, so even an account
holder can't read your AWS account — which is the right posture for a public app.

**Gotcha:** API methods that return `void` break the RPC client
(`decodeRpcResponse` → *"Cannot read properties of undefined (reading 'length')"*).
Return a small envelope like `{ ok: true }` from mutating methods (`sendMessage`,
`signOut`).

## 5. Debugging the deployed agent + upgrading a 0.x preview stack

### The bug: deployed agent fails on the first message ("S3 error reading …")

**Symptom:** in the cloud (sandbox) deploy, the first message returned
`⚠️ S3 error reading <sessionId>/scopes/agent/agent/snapshots/snapshot_latest.json`.

**Why S3 is even involved:** Lambda is stateless — nothing survives between
invocations, and the agent runs async (SQS→Lambda), so turn 2 may hit a different
Lambda. So the agent serialises its Strands session state to `snapshot_latest.json`
in S3 (the `FileBucket`) after each turn and **loads it back at the start of the
next turn**. (DynamoDB holds the message log; S3 holds the agent's internal state.)
On the **first** turn there's no snapshot yet — the load should return "empty."

**Root cause (traced end to end) — a hardcoded `us-east-1` default:**
UI error → CloudWatch (`aws logs tail` the Handler Lambda) → confirmed the object is
a plain 404 (`s3api head-object`) and the Lambda role has correct S3 perms + region.
The error is thrown in `@strands-agents/sdk` `s3-storage.js`: `_readJSON` only treats
`name==='NoSuchKey'`/`NoSuchBucket` as "empty" and throws everything else. The real
cause is one line up — the S3 client:

```js
// Strands S3Storage constructor:
this._s3 = config.s3Client ?? new S3Client({ region: config.region ?? 'us-east-1' });
```

bb-agent calls `new S3Storage({ bucket: bucket.fullId })` — passing **no region and no
client** — so Strands' S3 client **hardcodes `us-east-1`**, but the bucket is in
`ap-northeast-1`. `GetObject` to the wrong-region endpoint returns **`PermanentRedirect`
(301)**, not `NoSuchKey` → it throws `S3 error reading …`.

Why this was sneaky: it only breaks **on AWS** (local dev uses `FileBucketSnapshotStorage`,
which resolves the name/region correctly), only in **non-`us-east-1` regions** (a Tokyo
deploy — a us-east-1 deploy would have worked by luck), and it's **identical in bb-agent
0.1.3 and 0.3.0**, so the version upgrade didn't fix it. Contrast: `bb-file-bucket` builds
its client as `new S3Client({ customUserAgent })` with **no** region, letting the SDK
auto-resolve from the Lambda's `AWS_REGION` — which is why *it* works.

**The fix (patch-package) — verified working:** pass the region to Strands so it stops
defaulting to us-east-1. Patch `node_modules/@aws-blocks/bb-agent/dist/agent.aws.js`:

```js
(bucket) => new S3Storage({ bucket: bucket.fullId, region: process.env.AWS_REGION })
```

Make it durable so it survives `npm install`:
```bash
npm i -D patch-package
# ...edit the file above...
npx patch-package @aws-blocks/bb-agent        # writes patches/@aws-blocks+bb-agent+0.3.0.patch
# add  "postinstall": "patch-package"  to package.json scripts
```
After redeploy, the first message worked: real Claude Sonnet 4.6 (Bedrock, Tokyo) replied
and correctly reported the tool's cloud stub.

**Validated against upstream — and already fixed.** This is a known bug, tracked as
[aws-blocks#120](https://github.com/aws-devtools-labs/aws-blocks/issues/120) and fixed
by [PR #137](https://github.com/aws-devtools-labs/aws-blocks/pull/137) (merged
2026-07-03), which makes the **identical change** we did — passing
`region: process.env.AWS_REGION` to `S3Storage` in `agent.aws.ts`, plus a regression
test. Reaching that fix by tracing the failure end to end — UI error → CloudWatch → IAM
→ S3 → dependency source → one line in `agent.aws.js` — rather than guessing, is the
whole point: a disciplined trace lands on the same answer AWS's own maintainers did.

**Why the patch is still here.** As of this writing the fix is **merged but not yet
released to npm** (latest `bb-agent` is `0.3.0`, published before the merge), and any
non-`us-east-1` deploy is broken on the very first message without it. So the committed
`patches/@aws-blocks+bb-agent+0.3.0.patch` keeps this app running in Tokyo — the
`postinstall: patch-package` hook re-applies it on every `npm install`, surviving fresh
clones and CI. **When a fixed version ships, upgrade `bb-agent` and delete the patch**
(`patch-package` warns when the patched code no longer matches, which is the signal to
drop it).

**Debugging techniques worth keeping:**
```bash
aws logs tail /aws/lambda/<fn> --region ap-northeast-1 --since 15m --format short
aws iam get-role-policy --role-name <role> --policy-name <p>   # check S3 actions/resources
aws s3api head-object --bucket <b> --key <k>                   # does the object exist?
grep -rn "<error string>" node_modules                        # trace error to its source
```

### Upgrading the preview stack — and the version-skew gotcha

Fix attempt: bump the meta-package, which pulls the coordinated set:
```bash
npm install @aws-blocks/blocks@^0.2.0   # → bb-agent 0.3.0, core 0.1.10
```

**Gotcha:** the deploy then failed at module load with
`SyntaxError: ... '@aws-blocks/hosting/adapters' does not provide an export named
'normalizeBasePath'`. Cause: `core` 0.1.10 imports a newer `hosting` symbol, but
`core` declared its `hosting` dependency as `*`, so npm left the **stale
`hosting` 0.1.2** in place. **In a 0.x preview, coordinated releases can leave
sibling packages skewed; fix by pinning the lagging one:**
```bash
npm install @aws-blocks/hosting@^0.1.4
```

### `cdk synth` — validate before you deploy

Instead of discovering skews/errors during a slow, resource-creating deploy, run a
**local synth first**. It loads the entire module graph and builds the CloudFormation
template but creates **zero AWS resources** — a fast, free pre-flight:
```bash
npx cdk synth --quiet \
  --context sandboxMode=true --context projectRoot="$PWD" \
  --app "npx tsx -C cdk aws-blocks/index.cdk.ts"
```
If synth is clean, the deploy will get past the build/module-load stage.

## 6. The RAG pivot — "Ask my handover docs"

> For the architecture reference (services, index/query flow, S3 Vectors vs
> OpenSearch, key files) see [How RAG is implemented](./rag-implementation.md).
> This section is the *story* of the pivot and the gotchas hit along the way.

### Why we pivoted (and why the app never got write access)

The original ask was to give the deployed agent real powers ("actually do things
for me") by granting it PowerUser/SystemAdministrator. Rejected, with reasons worth
keeping: a **publicly reachable AI agent with write access to a personal AWS
account** is a cost-bomb + prompt-injection + resource-destruction risk — anyone
with the invite code (or a leaked one) could talk the model into creating or
deleting resources. Read-only was the safe ceiling for a public demo.

Instead the app pivoted to the exercise's original idea: **RAG (Retrieval-Augmented
Generation)** over a set of handover documents — "ask my docs" instead of "ask my
AWS account" — something a team can query while the author is away.

### What changed in the code

- `aws-blocks/aws-mcp.ts` + `scripts/smoke-mcp.ts` **deleted** (the MCP/AWS tool is gone).
- `aws-blocks/index.ts`: added a `KnowledgeBase` block and a `searchDocs` tool:

```ts
const kb = new KnowledgeBase(scope, 'docs', {
  source: './knowledge',        // local folder, uploaded to S3 on deploy
  removalPolicy: 'destroy',     // exercise stack — clean teardown
});
```

- System prompt rewritten for RAG discipline: **search first, answer only from
  retrieved passages, cite the source file, say "not found" otherwise.**
- Frontend re-themed ("Ask my handover docs").
- The e2e tests survived almost untouched — they test *flows* (signup with invite
  code, send a message, get a reply), not implementation details, so swapping the
  agent's tool didn't break them. That's the payoff of writing e2e tests against
  behavior instead of internals.

### Type gotcha: tool results must be `JSONValue`

Returning `kb.retrieve(query)` directly from the tool handler fails to compile:
`RetrieveResult[]` is not assignable to `JSONValue` (the interface has no index
signature). Map to a plain shape instead:

```ts
const results = await kb.retrieve(query);
return results.map((r) => ({ text: r.text, source: r.source, score: r.score }));
```

General lesson: **tool handlers return data that gets serialised into the model's
context** — keep returns as plain JSON-shaped objects, not SDK types.

### KnowledgeBase: local vs cloud are different machines

| | Local dev (`npm run dev`) | Cloud (deployed) |
|---|---|---|
| Search index | **TF-IDF** (free, instant, no ML) | **Titan v2 embeddings** + S3 Vectors |
| Matching | Literal keywords only | Semantic (synonyms/paraphrase match) |
| File formats | `.md .txt .html .htm .csv .json` only — **PDFs are silently skipped** (a `console.warn` in the dev terminal is the only hint) | Bedrock parses binaries too: `.pdf .doc .docx .xls .xlsx` |
| Ingestion | Instant at startup, cached in `.bb-data/<id>/chunks.json`, auto-invalidated by content hash | **Asynchronous after deploy** — `retrieve()` returns empty until sync completes (minutes); check `kb.isSynced()` / `kb.waitUntilSynced()` |
| Idle cost | Free | S3 Vectors is serverless — no idle compute; you pay embedding calls at ingestion + storage |

Two practical consequences:

1. **Don't judge retrieval quality locally.** TF-IDF only matches literal words
   ("who approves deploys?" won't find "sign-off from Priya for production
   releases"). Local dev proves the plumbing; the cloud judges quality.
2. **The post-deploy "it's broken" moment is expected.** First queries after
   `npm run deploy` return nothing until Bedrock finishes chunking + embedding.
   Wait, or poll `isSynced()`.

To test PDF-ish content locally anyway, convert first (`pdftotext -layout in.pdf
out.txt` — `-layout` preserves table columns, which matters for statements).

### TF-IDF in one paragraph

**T**erm **F**requency – **I**nverse **D**ocument **F**requency: classic pre-ML
ranking. A chunk scores high if the query's words appear often in it (term
frequency), weighted by how *rare* each word is across all chunks (inverse document
frequency) — so "rollback" and "Priya" count for a lot, "the" and "account" for
almost nothing. Pure word counting: free, instant, no model — which is exactly why
the local mock uses it.

### Privacy guard: `knowledge/` and a public repo

This repo is public, and `knowledge/` holds real documents. `.gitignore` uses:

```gitignore
knowledge/*
!knowledge/SAMPLE-handover.md
```

Note the form: `knowledge/*` (ignore *contents*), not `knowledge/` (ignore the
*directory*) — git cannot re-include a file inside an ignored directory, so the
`!` exception only works with the first form. The sample doc stays tracked so a
fresh clone works out of the box; everything else stays local.

## 7. Security remediation pass (2026-07-13)

After the app was live, a second security QA pass (four prompts: security headers,
OWASP review, credential-leak scan, exposed-API-key scan) produced a set of confirmed
findings. Those findings were filed as GitHub issues, and this pass implemented the
subset that is pure application code plus one CloudFront config change. The billable
controls (WAF, Cognito threat protection) were deliberately left for a later, cost-gated
change.

Good news first: no exposed API key, AWS credential, private key, or credible secret
was found anywhere — not in source, tracked files, git history, the live JS bundle
(~537 KB), the public `/.blocks-sandbox/config.json` (28 bytes, only `apiUrl`), or
`localStorage` (theme + conversation UUID only). Cognito tokens never touch the browser;
the session cookie is an opaque, `HttpOnly`/`Secure`/`SameSite=Lax` pointer. So findings
3 and 4 of the QA (leak checks) passed clean. The rest were hardening gaps.

### What was fixed

**Realtime channel authorization (the real bug).** Every conversation route derived
`userId` from the server-side session *and* verified ownership via `assertOwns()` — every
route except `getChannel()`, which checked only "are you signed in?". Because the Realtime
channel id **is** the conversation UUID (`src/App.tsx` `subscribe()`), any signed-in user
who learned another user's conversation UUID could mint a signed WebSocket subscription
token and read that user's streamed replies. This is a classic BOLA / IDOR: authentication
(*who are you*) is not authorization (*may you touch this object*). Object-level ownership
must be checked on **every** route that takes a client-supplied object id — no exceptions.

The fix (`aws-blocks/index.ts`, `getChannel`) is one line of enforcement:

```ts
await assertOwns(agent, user.username, channelId);
```

A permanent regression test was added (`test/e2e.test.ts`): User A creates a conversation,
User B signs in and must be *rejected* when calling `getChannel(conversationId)`. The test
was proven to have teeth by temporarily removing the guard and watching it fail with
`Missing expected rejection: cross-user realtime channel must be denied`, then restoring it.
Model-switching cannot bypass the check: channels are per-agent, so a mismatched model just
fails to find the conversation and still rejects.

**Input / cost guards.** Two cheap bounds against a single request running up an unbounded
Bedrock token bill (`aws-blocks/index.ts`):

- `sendMessage` now rejects empty and oversized messages (8 000-char cap) *before* the
  ownership lookup or Bedrock call.
- The `searchDocs` tool clamps `maxResults` to `[1, 10]` (the underlying block otherwise
  permits up to 100), so a model- or client-supplied value can't request an unbounded
  retrieval.

**Content-Security-Policy (`aws-blocks/index.cdk.ts`).** The live site relied on the AWS
managed SecurityHeadersPolicy, which intentionally ships no CSP. A CSP is the browser's
allowlist for scripts/styles/connections — the second line of defense if an XSS payload
ever lands. Rather than paste a generic CSP, each directive was validated against the
actual `npm run build` output, which let it be *stricter* than a naive draft:

- `script-src 'self'` with **no `'unsafe-inline'` and no `'unsafe-eval'`** — verified the
  bundle has zero inline scripts and zero `eval`/`Function`. This is the highest-value CSP
  directive.
- `connect-src 'self' https: wss:` kept permissive on purpose — the Realtime subscription
  connects to a cross-origin AppSync endpoint (https handshake + wss stream); a `wss:`-only
  policy would break streaming. A future tightening is to pin that exact endpoint host.
- `style-src 'self' 'unsafe-inline'` — the built `index.html` has one inline `<style>` and
  GSAP animates via inline style attributes.
- `base-uri`/`object-src 'none'`/`frame-ancestors 'none'`/`form-action 'self'`/
  `upgrade-insecure-requests` as defense-in-depth.

CSP is a CloudFront header, so it only takes effect after `npm run deploy`; verify in the
browser (DevTools console: no CSP violations, chat still streams) before trusting it.
`Permissions-Policy` is still missing — `Hosting` has no prop for it, so it needs a separate
custom CloudFront response-headers policy (follow-up).

### Inspecting & operating the deployed stack

Replace `<stack-name>` and `<region>` with your values — for this project they are
`ask-aws-agent-stack-prod` and `ap-northeast-1`.

List every resource AWS Blocks deployed, grouped by type (a handful of block
constructors expands to ~117 CloudFormation resources):

```bash
aws cloudformation list-stack-resources --stack-name <stack-name> --region <region> \
  --query "StackResourceSummaries[].ResourceType" --output text \
  | tr '\t' '\n' | sort | uniq -c | sort -rn
```

The **invite code** lives encrypted in SSM Parameter Store (a `SecureString` created by
the secret `AppSetting`) — never in Git or the frontend. You are its source of truth;
share it only with people you want to grant access. Read or rotate it:

```bash
# Read the current code
aws ssm get-parameter --name /ask-aws/invite-code --with-decryption \
  --region <region> --query "Parameter.Value" --output text

# Rotate it (instant — the Lambda reads SSM per request, no redeploy needed)
aws ssm put-parameter --name /ask-aws/invite-code --value <new-code> \
  --type SecureString --overwrite --region <region>
```
